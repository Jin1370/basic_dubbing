# 모바일 더빙 앱 QA 스킬

## 검증 철학

"존재 확인"이 아니라 **"경계면 교차 비교"**가 핵심이다. 각 컴포넌트가 개별적으로 올바르더라도, 연결 지점에서 계약이 어긋나면 런타임에 실패한다.

## 검증 영역

### 1. API 타입 정합성

프론트엔드 TypeScript 타입과 백엔드 Pydantic 모델을 직접 대조한다.

```
검증 절차:
1. server/models/schemas.py에서 각 응답 모델의 필드명과 타입 추출
2. lib/types.ts에서 대응하는 TypeScript 타입의 필드명과 타입 추출
3. 필드명 일치 여부 확인 (camelCase vs snake_case 변환 포함)
4. 타입 호환성 확인 (str↔string, int↔number, Optional↔undefined 등)
5. 래핑 구조 확인 (백엔드가 {data: [...]}를 반환하면 프론트가 .data를 접근하는지)
```

### 2. API 경로 정합성

```
검증 절차:
1. server/routers/에서 모든 @app.get/post/put/delete 데코레이터의 경로 추출
2. lib/api.ts에서 모든 axios 호출의 URL 추출
3. 1:1 매핑 확인 — 프론트가 호출하는 모든 API가 백엔드에 존재하는지
4. HTTP 메서드 일치 확인
```

### 3. 인증 흐름 검증

```
검증 절차:
1. 프론트의 Axios 인터셉터에서 Authorization 헤더 첨부 방식 확인
2. 백엔드의 auth 미들웨어에서 토큰 검증 방식 확인
3. 둘이 동일한 토큰 형식(Bearer JWT)을 사용하는지 확인
4. 인증이 필요한 엔드포인트 목록과 프론트의 인증 상태 관리가 일치하는지 확인
```

### 4. 더빙 상태 흐름 검증

```
검증 절차:
1. 백엔드의 더빙 상태 값(pending/processing/completed/failed) 추출
2. 프론트의 상태 표시 로직에서 처리하는 상태 값 추출
3. 모든 상태가 양쪽에서 처리되는지 확인
4. 상태 전이 순서가 논리적인지 확인
```

### 5. 네비게이션 경로 검증

```
검증 절차:
1. app/ 디렉토리의 파일 구조에서 Expo Router 경로 추출
2. 코드 내 router.push(), router.replace(), href 값 수집
3. 모든 링크가 실제 화면 파일과 매칭되는지 확인
```

## 보고서 형식

각 검증 항목마다:
- **상태**: PASS / FAIL / WARN
- **위치**: 파일 경로 + 라인 번호
- **기대값**: 프론트/백엔드 중 한쪽의 정의
- **실제값**: 다른 쪽의 정의
- **심각도**: Critical / Major / Minor
